# AutoJudge 통합 테스트 케이스

이 문서는 AutoJudge 법률 사건 분석 시스템의 통합 테스트를 위한 샘플 사건들을 정의합니다.

---

## 테스트 시나리오 개요

| 시나리오 | 설명 |
|---------|------|
| **기본 플로우** | Input Form → API → 결과 확인 |
| **What-If 분석** | 변수 변경 시나리오 비교 |
| **에러 핸들링** | 잘못된 입력, 누락된 데이터 처리 |

---

## 샘플 사건 1: 손핵배상 (교통사고)

### 기본 정보
| 항목 | 내용 |
|-----|------|
| **사건 유형** | 손핵배상 |
| **원고** | 김철수 (운전자 A) |
| **피고** | 박영희 (운전자 B) |

### 사실관계
2024년 3월 15일 오후 2시경, 서울시 강남구 테헤란로에서 원고 김철수는 직진 신호를 받고 교차로를 통과하던 중, 좌회전 신호를 위반한 피고 박영희의 차량과 충돌하였습니다. 이 사고로 원고의 차량 앞부분이 파손되었고, 원고는 경추 염좌로 3주간 치료를 받았습니다.

### 청구 내용
- 차량 수리비: 3,500,000원
- 치료비: 1,200,000원
- 통원치료 교통비: 150,000원
- 업무상 손해 (휴업 손해): 2,000,000원
- **총 청구액: 6,850,000원**

### 증거
1. **CCTV 영상** - 교차로 CCTV에 충돌 장면이 녹화됨
2. **교통사고사실확인원** - 경찰서 발급, 피고의 신호위반 사실 기재
3. **진단서** - 병원 발급, 경추 염좌 진단
4. **수리비 견적서** - 자동차 정비소 발급
5. **소득증명원** - 원고의 직장 발급 (월급 400만원)

### 예상 시나리오
| 시나리오 | 승소 가능성 | 핵심 쟁점 |
|---------|-----------|----------|
| **최상 시나리오** | 높음 (85%) | CCTV와 교통사고사실확인원으로 피고의 전 과실이 명확 |
| **중간 시나리오** | 중간 (60%) | 원고도 일부 과실 있음 (과실상계 20% 적용 가능) |
| **최하 시나리오** | 낮음 (30%) | 증거 불충분, 피고가 원고의 과실 주장 |

### What-If 테스트

#### 변경 시나리오 A: CCTV 증거 없음
- **변경사항**: CCTV 영상을 증거에서 제외
- **예상 영향**: 승소 가능성 하락 (CCTV 없이는 신호위반 입증 어려움)

#### 변경 시나리오 B: 청구액 증가
- **변경사항**: 휴업 손해를 2,000,000원 → 5,000,000원으로 증가
- **예상 영향**: 총 청구액 증가, but 과다 청구로 인한 신뢰도 하락 가능성

## 샘플 사건 2: 계약위반 (임대차)

### 기본 정보
| 항목 | 내용 |
|-----|------|
| **사건 유형** | 계약위반 |
| **임대인 (원고)** | 이상호 |
| **임차인 (피고)** | 최민지 |

### 사실관계
2023년 1월 1일, 원고 이상호와 피고 최민지는 서울시 마포구 아파트에 대해 2년 임대차계약을 체결하였습니다. 보증금은 5천만원, 월세는 100만원으로 정하였습니다. 계약서에는 "임차인은 임대차 목적물을 주거 외의 목적으로 사용할 수 없다"는 조항이 포함되어 있었습니다.

2024년 2월, 원고는 피고가 해당 주택을 불법 숙박업소로 운영하고 있다는 민원을 접수하고 현장을 확인하였습니다. 피고는 에어비앤비 플랫폼을 통해 단기 임대를 하고 있었으며, 1년간 약 500만원의 수익을 올린 것으로 확인되었습니다.

### 청구 내용
1. **계약 해지** - 계약위반으로 인한 임대차계약 해지
2. **보증금 반환 거부** - 위약금으로 보증금 5천만원 중 1천만원 공제
3. **손핵배상** - 불법 영업으로 인한 명예훼손 및 건축법 위반으로 인한 손해: 500만원
4. **명도 청구** - 즉시 주택 인도 요구

### 증거
1. **임대차계약서** - 주거 외 사용 금지 조항 포함
2. **에어비앤비 예약 내역 스크린샷** - 피고의 숙박업 운영 증거
3. **수익 정산 내역** - 플랫폼에서의 수익 내역
4. **관할 주민센터 민원 접수증** - 불법 숙박업 신고 내역
5. **동대표 확인서** - 건축물 위반 사실 확인

### 예상 시나리오
| 시나리오 | 승소 가능성 | 핵심 쟁점 |
|---------|-----------|----------|
| **최상 시나리오** | 높음 (80%) | 명백한 계약위반, 증거 확실 |
| **중간 시나리오** | 중간 (55%) | 피고가 "동의 있었다" 주장, 사실관계 다투어짐 |
| **최하 시나리오** | 낮음 (25%) | 피고가 주거 목적 준수 주장, 에어비앤비가 가끔의 친구 초대라 주장 |

### What-If 테스트

#### 변경 시나리오 A: 구두 허락 존재
- **변경사항**: 원고가 "단기 렌트 허락함"이라는 문자메시지 추가
- **예상 영향**: 계약위반 성립 여부 불명확, 승소 가능성 급락

#### 변경 시나리오 B: 피고가 무단 퇴실
- **변경사항**: 피고가 보증금 반환 없이 2024년 3월에 퇴실함
- **예상 영향**: 명도 소송 불필요, 보증금 반환 청구소송으로 변경

## 샘플 사건 3: 부당이득

### 기본 정보
| 항목 | 내용 |
|-----|------|
| **사건 유형** | 부당이득 |
| **원고 (송금인)** | 정재훈 |
| **피고 (수취인)** | 한수민 |

### 사실관계
2024년 1월 10일, 원고 정재훈은 거래처 "한수산업"에 대금 10,000,000원을 송금하고자 하였습니다. 은행 앱에서 수취인을 검색하는 과정에서, 유사한 이름의 피고 한수민의 계좌를 실수로 선택하여 송금하였습니다.

원고는 즉시 은행에 연락하였으나, 피고 계좌는 이미 해지되었고 잔액이 인출된 상태였습니다. 원고는 피고에게 연락을 시도하였으나, 피고는 "돈을 받은 적 없다"며 부인하고 있습니다.

### 청구 내용
- **부당이득 반환 청구** - 10,000,000원 및 지연 이자 (연 5%)
- **소송 비용** - 피고 부담

### 증거
1. **송금 내역** - 은행 송금 확인증 (수취인: 한수민, 계좌번호: ***-***-******)
2. **원고-거래처 이메일** - "한수산업"과의 거래 내용 및 송금 요청
3. **통장 거래 내역** - 원고 계좌에서 1천만원 출금 기록
4. **피고 계좌 해지 내역** - 은행 발급, 송금일 다음 날 해지
5. **원고-피고 통화 녹취** - 피고가 수취 부인하는 내용

### 예상 시나리오
| 시나리오 | 승소 가능성 | 핵심 쟁점 |
|---------|-----------|----------|
| **최상 시나리오** | 높음 (90%) | 송금 내역 명확, 피고 해지 시점이 의심스러움 |
| **중간 시나리오** | 중간 (65%) | 피고가 "정당한 사유" 주장 (예: 원고와의 과거 거래 등) |
| **최하 시나리오** | 낮음 (40%) | 증거 부족, 피고가 계좌 미사용 주장 (해킹 등) |

### What-If 테스트

#### 변경 시나리오 A: 피고가 부분 반환
- **변경사항**: 피고가 500만원만 반환하고 나머지는 소비함
- **예상 영향**: 부분 승소, 나머지 500만원 반환 판결 예상

#### 변경 시나리오 B: 송금 1년 후 발견
- **변경사항**: 송금 실수를 1년 후에 발견 (소멸시효 문제)
- **예상 영향**: 부당이득 청구권 소멸시효(5년) 미경과로 청구 가능

## E2E 테스트 절차

### 테스트 환경 설정
```bash
# 1. 서버 시작
cd /root/projects/autojudge/backend
python -m uvicorn main:app --reload --port 8000

# 2. 테스트 실행
cd /root/projects/autojudge/backend
python -m pytest test_integration.py -v
```

### 테스트 시나리오

#### TC-001: 기본 사건 생성 및 시나리오 생성
1. POST /api/cases - 사건 생성
2. POST /api/scenarios - AI 시나리오 생성 (mocked)
3. GET /api/scenarios/{id} - 결과 조회
4. **검증**: 3개의 시나리오가 생성되고 각각 title, interpretation, basis, probability, key_fields를 포함

#### TC-002: What-If 시나리오 비교
1. POST /api/cases - 원본 사건 생성
2. POST /api/whatif - 변수 변경 시나리오 생성
3. GET /api/whatif/{id} - 비교 결과 조회
4. **검증**: original_case와 modified_case가 모두 존재하고, differences 배열에 변경사항이 기록됨

#### TC-003: 에러 핸들링 - 존재하지 않는 사건
1. POST /api/scenarios - 존재하지 않는 case_id로 요청
2. **검증**: HTTP 404 반환, "Case not found" 메시지 포함

#### TC-004: 에러 핸들링 - 잘못된 What-If 요청
1. POST /api/whatif - 변경사항 없이 요청 (changes에 아무 값도 없음)
2. **검증**: HTTP 400 반환, "At least one change must be specified" 메시지 포함

#### TC-005: 에러 핸들링 - 과도한 What-If 변경
1. POST /api/whatif - 3개 이상의 필드 변경 요청
2. **검증**: HTTP 400 반환, "Maximum 2 variables can be changed" 메시지 포함

## OpenRouter API Mocking

테스트에서 OpenRouter API 호출은 반드시 mocking되어야 합니다:

```python
# test_integration.py에서 사용할 mock 예시
@pytest.fixture
def mock_openrouter():
    with patch("services.scenario.OpenAI") as mock_client:
        mock_response = MagicMock()
        mock_response.choices[0].message.content = json.dumps({
            "scenarios": [
                {
                    "title": "시나리오 1",
                    "interpretation": "해석 내용...",
                    "basis": ["[law] 민법 제750조"],
                    "probability": "high",
                    "key_factors": ["핵심 쟁점 1"]
                },
                # ... more scenarios
            ]
        })
        mock_client.return_value.chat.completions.create.return_value = mock_response
        yield mock_client
```
